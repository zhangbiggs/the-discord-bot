# 异步接入问题的思路、代码组织与边界处理

## 一、核心异步思路

```
┌─ POST /interactions ──────────────────┐
│  type: 2 (APPLICATION_COMMAND)        │
│  /opus <youtube-url>                  │
└──────────────┬────────────────────────┘
               │
               ▼
┌─ Handler (3秒内必须返回) ───────────────┐
│  1. 解析 body (type, name, options)    │
│  2. 校验参数 (url 必填)                │
│  3. asyncio.create_task(后台流程)      │ ← 不 await！
│  4. return {"type": 5}                │ ← 立刻返回，显示 "is thinking…"
└────────────────────────────────────────┘
               │
               ▼
┌─ _process_opus() (后台任务) ──────────┐
│                                       │
│  ┌─ Phase3 ──────────────────────┐    │
│  │ GET /v1/transcript?url=...    │    │
│  │ POST /v1/chat/completions     │    │
│  │ → start_seconds / end_seconds │    │
│  └───────────────────────────────┘    │
│               │                       │
│               ▼                       │
│  ┌─ Phase1 ──────────────────────┐    │
│  │ POST /v1/clip                 │    │
│  │  +callback_url=/clips/done    │    │
│  │ → job_id                      │    │
│  │ store mapping job_id→{app,tok}│    │
│  └───────────────────────────────┘    │
│                                       │
└───────────────────────────────────────┘
               │  60秒后 (pipeline完成)
               ▼
┌─ POST /clips/done (回调) ─────────────┐
│  receive {job_id, video_url}          │
│  _pop_job() → 幂等消费                │
│  _store_token() → Phase2 token        │
│  POST /webhooks/{app_id}/{token}      │
│  → "🎬 Clip ready! claim/xxxx"        │
└────────────────────────────────────────┘
```

**关键决策**：选择 **webhook 回调**而非轮询。回调天然幂等（`_pop_job` 一次性消费），零浪费请求。

## 二、代码组织

单文件 `bot.py`，按职责从上到下分层：

| 区块 | 职责 |
|------|------|
| 配置 | 环境变量 + 默认值，适配 Docker/宿主机两种运行模式 |
| 存储层 | `_store_job`, `_pop_job`, `_store_token`, `_get_token`, `_claim_token` + `asyncio.Lock` |
| 应用生命周期 | `lifespan`: 创建/关闭全局 `httpx.AsyncClient`，绕过系统代理 |
| **Phase 1 入口** | `POST /interactions` — 纯编排，3 秒内返回 |
| **Phase 1 后台** | `_process_opus()` — 整个异步工作流编排 |
| **Phase 1 回调** | `POST /clips/done` — pipeline 完成回调 |
| 工具函数 | `_send_followup()` |
| **Phase 2 API** | `POST /api/tokens` + `GET /api/claim/{token}` |
| **Phase 3 LLM** | `_fetch_transcript()` + `_pick_window_with_llm()` |
| 健康检查 | `GET /healthz` |
| 入口 | `uvicorn.run()` |

**组织原则**：
- 入口函数只做**快速路由和校验**，不做任何 `await` 外部调用
- 所有耗时操作在 `asyncio.create_task` 的后台协程中完成
- LLM 逻辑（Phase 3）与 Token API（Phase 2）通过**存储层共享数据**，互不耦合

## 三、边界处理

### 1. Discord 3 秒超时 — 最核心的约束

```python
# ✅ 正确：create_task 调度后立刻返回
asyncio.create_task(_process_opus(url, app_id, token))
return {"type": 5}

# ❌ 错误：不能 await 任何外部调用
# await _process_opus(...)  # 这会阻塞直到 pipeline 完成
```

Handler 只做三件事：
1. 解析请求 body
2. 校验参数完整性
3. 返回 `{"type": 5}`（deferred 占位）

任何外部 API 调用、数据库写入、LLM 请求都必须在后台任务中执行。

### 2. Pipeline 超过 3 秒 — 通过 deferred + 回调解决

Discord 给出了 15 分钟的 follow-up 窗口：

```python
# 立即返回 type 5 占位 → Discord 显示 "is thinking…"
# 15 分钟内可通过 webhook 补充真正内容
POST /webhooks/{app_id}/{token}
{"content": "🎬 Your clip is ready!", ...}
```

第一次 follow-up **编辑替换**占位消息，后续 follow-up **追加**新消息。

### 3. LLM 对抗性输出 — 三层防御

mock-llm 默认是"对抗性"的（`LLM_ADVERSARIAL=1`），在 prompt 不够明确时返回散文而非 JSON。

```
第一层: response_format: {"type": "json_object"}
        + 系统 prompt 明确描述 JSON schema
        + prompt 中包含 "JSON", "structured", "schema" 等关键词
        → 触发 mock-llm 的 caller_asked_for_structure() 返回 True

第二层: JSONDecodeError 捕获
        → re.search(r'\{.*\}', content, re.DOTALL) 正则提取 JSON 对象

第三层: 全局 except → 中间段 fallback
        → start = duration / 3, end = min(start + 30, duration)
```

### 4. 幂等性

| 场景 | 机制 | HTTP 结果 |
|------|------|-----------|
| 重复 pipeline 回调 | `_pop_job()` 原子删除，第二次返回 `already processed` | 200 + warning |
| 重复 token 领取 | `_claim_token()` 检查 `claimed` 标志 | 409 Conflict |
| 过期 token | `_get_token()` 检查 `created_at + 24h` | 404 Not Found |

### 5. 网络异常处理

每一层外部调用都有独立的 try/except：

```python
# LLM 调用
try:
    result = await _pick_window_with_llm(client, transcript)
except Exception:
    # 降级：不传 start_seconds/end_seconds，让 pipeline 处理全片

# Pipeline 调用
try:
    resp = await client.post(f"{MOCK_PLATFORM}/v1/clip", ...)
except Exception as exc:
    # 发送错误消息到 Discord
    await _send_followup(client, app_id, token, f"❌ {exc}")

# Follow-up 发送
try:
    await _send_followup(...)
except Exception:
    # 即使 follow-up 失败也不能让整个请求失败
    logger.error("Fallback follow-up also failed: %s", e2)
```

### 6. 系统代理拦截

```python
# httpx 默认会读取系统代理设置 (如 ClashX 的 127.0.0.1:1082)
# 导致 localhost 请求被代理截获然后断开连接
_http = httpx.AsyncClient(
    mounts={
        "http://": httpx.AsyncHTTPTransport(),
        "https://": httpx.AsyncHTTPTransport(),
    },
)
```

通过显式配置 transports 绕过系统代理检测。

### 7. 运行时配置切换

所有服务 URL 通过环境变量配置，自动适配两种运行模式：

| 环境变量 | 宿主机默认值 | Docker 默认值 |
|----------|-------------|---------------|
| `MOCK_DISCORD_URL` | `http://localhost:7001` | `http://host.docker.internal:7001` |
| `MOCK_PLATFORM_URL` | `http://localhost:7002` | `http://host.docker.internal:7002` |
| `MOCK_LLM_URL` | `http://localhost:7003` | `http://host.docker.internal:7003` |
| `CALLBACK_HOST` | `http://host.docker.internal:8080` | `http://host.docker.internal:8080` |
| `BOT_PORT` | `8080` | `8080` |
